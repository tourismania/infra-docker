# Issue #22 — Watchdog не детектирует зависший getUpdates: getMe проходит, бот молчит (follow-up #16)

GitHub Issue: https://github.com/tourismania/infra-docker/issues/22

## Проблема

Watchdog, добавленный в #16, проверяет живость через `bot.get_me()`. 2026-07-17 бот снова завис: после единичного `ConnectTimeout` через xray (07:28 UTC) long-polling `getUpdates` полностью остановился, но `getMe` продолжал успешно проходить — watchdog каждые 3 минуты рапортовал «связь с Telegram API в порядке», рестарта не произошло, бот не отвечал пользователям.

Из логов: после 07:28 в httpx-логах есть только `POST .../getMe 200 OK` и ни одной строки `getUpdates` (при живом поллинге они появляются каждые ~10 сек). Контейнер `Up`, процесс жив.

## Причина

`getMe` и `getUpdates` используют разные HTTP-клиенты (у `get_updates` отдельный пул с `connection_pool_size=1`). Зависает именно соединение поллинга — watchdog проверяет не тот канал. Проверка `getMe` не покрывает сценарий «туннель для getUpdates мёртв, новое соединение для getMe устанавливается успешно».

## Acceptance Criteria

- [x] Watchdog проверяет живость самого поллинга: фиксируется timestamp последнего успешного `getUpdates` (обёртка над `HTTPXRequest` или аналог); если он старше N × интервал поллинга (например, 90–120 сек), проверка считается проваленной.
- [x] Существующий порог `WATCHDOG_MAX_FAILURES` подряд сохраняется — после его достижения процесс завершается через `os._exit(1)`, контейнер перезапускается политикой `restart: unless-stopped`.
- [x] Порог «свежести» getUpdates конфигурируется через переменную окружения с разумным дефолтом.
- [x] Логи явно показывают возраст последнего успешного `getUpdates` при каждой проверке watchdog.
- [x] Обновлена документация (`CLAUDE.md`), создан `docs/issues/22.md`.

## Negative Constraints

- Не трогать бизнес-логику опросника (`ConversationHandler`, тексты, порядок вопросов) — только сетевой/инфраструктурный слой.
- Не убирать существующую проверку `getMe` без необходимости — она полезна как индикатор общей доступности API.
- Не добавлять новые внешние зависимости (`requirements.txt`: `python-telegram-bot[job-queue]`).
- Не рестартовать по единичному блипу — порог из нескольких проваленных проверок подряд обязателен.
- Поддержка работы без прокси (`XRAY_HTTP_PROXY=` пустой) должна сохраниться.

## Реализация

- `services/telegram-bot/telegram_bot.py`:
  - `_GetUpdatesFreshnessTracker(HTTPXRequest)` — переопределяет `do_request`, фиксируя `time.monotonic()` в модульной переменной `_last_get_updates_at` при каждом завершившемся (HTTP 200) запросе. Класс используется исключительно для `getUpdates`, поэтому его таймстемп не путается с `getMe` и другими методами бота, у которых свой, отдельный `HTTPXRequest`.
  - В `main()` инстанс трекера подключается через `ApplicationBuilder.get_updates_request(...)` вместо прежних `.get_updates_connect_timeout()/.get_updates_read_timeout()/.get_updates_pool_timeout()` — ApplicationBuilder не позволяет смешивать готовый request-инстанс с этими билдерами, поэтому таймауты (10/30/10, без изменений) переехали в конструктор трекера. Проксирование через xray по-прежнему берётся из `HTTP_PROXY`/`HTTPS_PROXY` (env), явный `proxy=` не передаётся — поведение без прокси не изменилось.
  - `watchdog_check` теперь помимо `get_me()` считает возраст `_last_get_updates_at` и сравнивает с `WATCHDOG_GETUPDATES_MAX_AGE_SECONDS`; оба сигнала (ошибка `get_me` или устаревший `getUpdates`) инкрементируют один и тот же счётчик `watchdog_failures`, так что порог `WATCHDOG_MAX_FAILURES` и защита от единичного блипа не изменились. В лог на каждой проверке пишется возраст последнего `getUpdates` в секундах.
  - Новая переменная окружения `WATCHDOG_GETUPDATES_MAX_AGE_SECONDS` (по умолчанию `120`).
- `CLAUDE.md` — обновлена заметка про сервис `telegram-bot`.

**Проверено локально:** воспроизведён ровно тот сценарий из issue (успешный `get_me`, устаревший `getUpdates`) с помощью юнит-прогона `watchdog_check` — после `WATCHDOG_MAX_FAILURES` подряд неудачных проверок вызывается `os._exit(1)`; на «здоровом» тике счётчик неудач сбрасывается в 0.
