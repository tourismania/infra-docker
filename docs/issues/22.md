# Issue #22 — Watchdog не детектирует зависший getUpdates: getMe проходит, бот молчит (follow-up #16)

GitHub Issue: https://github.com/tourismania/infra-docker/issues/22

## Проблема

Watchdog, добавленный в #16, проверяет живость через `bot.get_me()`. 2026-07-17 бот снова завис: после единичного `ConnectTimeout` через xray (07:28 UTC) long-polling `getUpdates` полностью остановился, но `getMe` продолжал успешно проходить — watchdog каждые 3 минуты рапортовал «связь с Telegram API в порядке», рестарта не произошло, бот не отвечал пользователям.

Из логов: после 07:28 в httpx-логах есть только `POST .../getMe 200 OK` и ни одной строки `getUpdates` (при живом поллинге они появляются каждые ~10 сек). Контейнер `Up`, процесс жив.

## Причина

`getMe` и `getUpdates` используют разные HTTP-клиенты (у `get_updates` отдельный пул с `connection_pool_size=1`). Зависает именно соединение поллинга — watchdog проверяет не тот канал. Проверка `getMe` не покрывает сценарий «туннель для getUpdates мёртв, новое соединение для getMe устанавливается успешно».

## Пересмотр решения (code review PR #23)

Первая версия PR добавляла проверку свежести `getUpdates` **вдобавок** к существующей `get_me()` — оба сигнала считались в общий счётчик неудач. В ревью (kirin-bear) справедливо указано: раз мы уже проверяем именно тот канал, который нужен (`getUpdates`), отдельная проверка `get_me()` ничего не добавляет — она была лишь суррогатом для "API вообще доступен", введённым в #16 до того, как появился прямой способ проверить сам поллинг. Если `getUpdates` регулярно завершается (даже без новых апдейтов, long-poll всё равно отвечает), это уже доказывает доступность API; если завис — не важно, отвечает ли `get_me()` из отдельного соединения, бот всё равно не видит сообщений пользователей.

Решение: `get_me()` из watchdog убран, единственный сигнал — возраст последнего успешного `getUpdates`. Это одновременно проще (один сигнал, меньше кода, не нужен `asyncio.wait_for`/`WATCHDOG_TIMEOUT_SECONDS`) и точнее (нет случая, когда `get_me()` маскирует зависший поллинг, как в исходном инциденте).

## Реализация

- `services/telegram-bot/telegram_bot.py`:
  - `_GetUpdatesFreshnessTracker(HTTPXRequest)` — переопределяет `do_request`, фиксируя `time.monotonic()` в модульной переменной `_last_get_updates_at` при каждом завершившемся (HTTP 200) запросе. Класс используется исключительно для `getUpdates` — в `main()` он подключается через `ApplicationBuilder.get_updates_request(...)`, который по устройству PTB и раньше был отдельным HTTP-клиентом от общего (см. `Причина`), так что это не третье соединение, а замена уже существующего дефолтного объекта на инстанс, умеющий сообщать нам о своей активности.
  - `.get_updates_request(...)` заменил прежние `.get_updates_connect_timeout()/.get_updates_read_timeout()/.get_updates_pool_timeout()` — ApplicationBuilder не даёт смешивать готовый request-инстанс с этими билдерами, поэтому таймауты (10/30/10, без изменений) переехали в конструктор трекера. Прокси по-прежнему берётся из `HTTP_PROXY`/`HTTPS_PROXY` (env), явный `proxy=` не передаётся — поведение без прокси не изменилось.
  - `watchdog_check` проверяет только возраст `_last_get_updates_at` против `WATCHDOG_GETUPDATES_MAX_AGE_SECONDS`; `get_me()`-проверка и связанный с ней `WATCHDOG_TIMEOUT_SECONDS` удалены как избыточные (см. `Пересмотр решения`). Порог `WATCHDOG_MAX_FAILURES` и защита от единичного блипа не изменились. В лог на каждой проверке пишется возраст последнего `getUpdates` в секундах.
  - Новая переменная окружения `WATCHDOG_GETUPDATES_MAX_AGE_SECONDS` (по умолчанию `120`). `WATCHDOG_TIMEOUT_SECONDS` удалена как неиспользуемая.
- `CLAUDE.md` — обновлена заметка про сервис `telegram-bot`.

**Проверено локально:** воспроизведён ровно тот сценарий из issue (устаревший `getUpdates`) с помощью юнит-прогона `watchdog_check` против реально установленного `python-telegram-bot[job-queue]==21.9` — после `WATCHDOG_MAX_FAILURES` подряд неудачных проверок вызывается `os._exit(1)`; на «здоровом» тике счётчик неудач сбрасывается в 0.
